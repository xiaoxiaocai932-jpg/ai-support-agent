# 阶段路线与完成状态

## 开发总原则

每个新模块开发前，必须先确认以下内容：

```text
1. 目标用户是谁
2. 用户在什么场景下使用
3. 这个页面在主流产品中通常是什么形态
4. 页面应该展示什么
5. 页面不应该展示什么
6. 用户完成任务的主流程是什么
7. AI 在这个流程中负责判断、生成、推荐还是执行
8. 哪些动作需要人工确认
9. 本阶段验收标准是什么
10. 用户确认后再写代码
```

## 客户补充说明自动续单完成记录

```text
问题等级：P1 主链路体验问题

问题：
客户收到客服回复后，如果问题仍未解决但没有点击“仍需帮助”，而是直接继续输入消息，原流程没有明确处理方式。

修正：
1. 当工单处于“待客户确认”时，客户继续输入消息默认作为当前工单补充说明
2. 系统不再弹出“补充当前工单 / 作为新问题咨询”的二次选择
3. 补充说明保存到 SQLite
4. 工单状态从“待客户确认”回到“处理中”
5. 客服工作台展示客户补充说明
6. 客服回复草稿生成时会带入客户补充说明

产品意义：
客户侧不承担工单归属判断，系统根据未闭环上下文自动续单，更符合主流客服产品的低打扰使用习惯。

修复后原计划：
继续进入线上部署与交付说明。
```

## 客户侧提交工单入口修正记录

```text
问题等级：P1 主链路或可用性严重问题

问题：
客户侧 AI 回答后，“建议转人工处理”“仍需帮助”“确认提交工单”同时出现，用户容易分不清系统建议、用户反馈和业务动作的区别。

修正：
1. AI 判断需要人工时，只显示轻提示，不直接展示提交工单入口
2. 客户先选择“已解决 / 仍需帮助”
3. 只有客户点击“仍需帮助”后，才展示提交工单提示和“确认提交工单”按钮
4. “仍需帮助”只记录反馈并引导下一步，“确认提交工单”才真正创建工单

产品意义：
修正后客户侧流程更符合主流客服产品逻辑，减少误操作和理解成本。

修复后原计划：
继续进入知识库管理后台。
```
