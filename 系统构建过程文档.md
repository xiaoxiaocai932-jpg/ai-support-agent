# 智能客户支持 Agent 平台系统构建过程文档

## 1. 文档用途

这份文档用于记录智能客户支持 Agent 平台从 0 到 1 的系统建设过程。

它重点说明：

```text
1. 项目的业务背景和建设目标
2. 每个阶段解决了什么业务问题
3. 每个阶段的目标用户是谁
4. 每个阶段做了哪些功能
5. AI 在每个阶段承担什么职责
6. 人工在每个阶段保留什么确认权
7. 每个阶段如何验收
8. 每个阶段产生了什么业务价值和系统建设价值
9. 当前已完成什么，后续还要补什么
```

## 2. 项目定位

项目名称：

```text
智能客户支持 Agent 平台
```

行业场景：

```text
B2B SaaS 企业客户支持
```

项目方向：

```text
企业级 AI Agent / 企业知识库 + 工作流自动化
```

项目目标：

```text
客户可以用自然语言提问，AI 基于企业知识库回答；
AI 无法解决或风险较高的问题，可以进入工单流程；
客服可以基于 AI 辅助信息处理问题；
运营可以维护知识库；
产品经理可以通过数据看板评估 AI 效果。
```

## 3. 业务背景与建设必要性

B2B SaaS 客户支持场景通常存在大量重复咨询和复杂问题流转，适合通过企业知识库和 AI Agent 提升服务效率：

```text
1. 有明确业务场景，不是泛聊天工具
2. 有企业知识库，可以为 AI 回答提供依据
3. 有客户问题和人工处理，适合建设自动化流转能力
4. 有高风险问题，适合讲人工兜底和 AI 边界
5. 有反馈和数据看板，可以持续评估服务效果
6. 能通过知识运营持续提升 AI 回答质量
```

## 4. 系统角色

| 角色 | 使用页面 | 核心诉求 |
| --- | --- | --- |
| 客户侧使用者 | 客户侧 AI 帮助中心 | 快速获得答案，未解决时能转人工 |
| 客服 / 技术支持 | 客服工单工作台 | 理解客户问题，处理工单，回复客户 |
| 运营 / 知识库管理员 | 知识库管理后台 | 维护知识内容，提高 AI 回答质量 |
| 产品经理 / 主管 | 数据看板 | 评估 AI 效果，发现优化方向 |

## 5. 总体业务闭环

```text
客户提问
↓
系统检索企业知识库
↓
AI 基于知识库生成回答
↓
客户反馈是否解决
↓
未解决时客户确认提交工单
↓
客服在工作台查看工单
↓
AI 辅助客服总结问题、判断分类、生成回复草稿
↓
客服人工确认并回复客户
↓
客户查看处理结果
↓
反馈和工单数据进入数据看板
↓
运营根据数据优化知识库
```

## 7. 阶段一：客户侧 AI 帮助中心

### 7.1 阶段目标

让客户可以像使用主流智能客服一样，用自然语言提问，并获得基于企业知识库的回答。

### 7.2 目标用户

```text
B2B SaaS 客户公司的普通使用者、管理员、财务、运营人员。
```

### 7.3 真实使用场景

客户在使用 SaaS 产品时遇到账号、发票、数据同步、套餐计费等问题，希望快速得到答案，不希望理解内部分类、优先级、负责团队等客服后台信息。

### 7.4 已实现功能

```text
1. 聊天式提问入口
2. AI 回答展示
3. 引用来源展示
4. 已解决 / 仍需帮助反馈
5. 客户确认后提交工单
6. 工单提交后查看状态
7. 客户侧查看客服回复
```

### 7.5 AI 职责

```text
1. 理解客户问题
2. 检索知识库
3. 基于知识库生成回答
4. 判断是否可能需要人工确认
```

### 7.6 人工职责

```text
客户决定回答是否解决问题；
客户确认是否提交工单；
客服负责后续处理和回复。
```

### 7.7 关键产品判断

客户侧不能展示过多内部信息，例如内部负责团队、详细置信度分数、复杂分类规则、模型参数和检索分数。

这样做的原因是：客户只关心问题是否被解决，不需要理解企业内部如何分派和判断。

### 7.8 验收标准

```text
1. 客户能输入问题并获得回答
2. 回答必须展示引用来源
3. 客户能反馈已解决或仍需帮助
4. 只有客户点击仍需帮助后，才展示提交工单入口
5. 客户提交工单后，客服工作台能看到该工单
6. 客户能看到客服后续回复
```

### 7.9 阶段产出价值

```text
1. 客户侧形成了从提问、获得回答到转人工的完整入口。
2. AI 回答通过引用来源增强了可信度，避免客户只看到无依据结论。
3. 页面隐藏内部分类、优先级和负责团队，降低了客户理解成本。
4. “仍需帮助”与“确认提交工单”分步出现，减少了误操作和流程混淆。
```

## 8. 阶段二：客服工单工作台

### 8.1 阶段目标

当 AI 无法解决客户问题时，让问题进入人工处理流程，并让客服能基于 AI 辅助信息快速处理。

### 8.2 目标用户

```text
客服人员、技术支持、客户成功人员。
```

### 8.3 真实使用场景

客户点击“仍需帮助”并确认提交工单后，客服需要查看客户原始问题、AI 已回答内容、引用知识、问题分类、优先级和建议负责团队。

### 8.4 已实现功能

```text
1. 工单列表
2. 工单详情
3. 客户原始问题
4. AI 已回复内容
5. AI 辅助判断
6. 引用知识
7. 工单状态流转
8. 客户反馈同步展示
```

### 8.5 AI 职责

```text
1. 总结客户问题
2. 推荐问题分类
3. 推荐优先级
4. 推荐负责团队
5. 展示引用知识
```

### 8.6 人工职责

```text
1. 判断是否接手处理
2. 修改工单状态
3. 决定是否回复客户
4. 对最终服务结果负责
```

### 8.7 关键产品判断

AI 可以辅助客服，但不能直接替代客服关闭工单或承诺处理结果。

原因是工单状态代表真实服务进度，涉及客户感知和企业责任，必须保留人工确认。

### 8.8 验收标准

```text
1. 客户提交的工单能进入客服工作台
2. 客服能查看客户问题和 AI 回答
3. 客服能查看 AI 分类、优先级、负责团队和引用知识
4. 客服能修改工单状态
5. 客户侧反馈能同步到客服工作台
```

### 8.9 阶段产出价值

```text
1. 客户问题从单次对话进入可跟踪的工单流程。
2. 客服可以看到 AI 总结、分类、优先级和引用知识，减少人工分拣成本。
3. 工单状态由客服人工确认，避免 AI 直接改变真实服务进度。
4. 客服侧承接了客户侧不应展示的内部处理信息，角色边界更清晰。
```

## 9. 阶段三：工单回复与数据持久化

### 9.1 阶段目标

补齐从“客户提交工单”到“客服回复客户”的服务闭环，并保证工单、回复、状态不会因服务重启丢失。

### 9.2 目标用户

```text
客服人员、客户侧使用者、系统维护者。
```

### 9.3 已实现功能

```text
1. 客服回复编辑框
2. AI 推荐回复草稿
3. 客服人工确认后发送回复
4. 回复记录展示
5. 客户侧刷新工单进展
6. 客户侧查看客服回复
7. 客户侧确认客服回复是否解决
8. 工单回复后进入待客户确认状态
9. 客户在待确认期间继续输入时，自动补充到当前工单
10. 客服工作台展示客户补充说明
11. SQLite 保存工单
12. SQLite 保存回复
13. SQLite 保存客户补充说明
14. SQLite 保存工单状态
15. SQLite 保存客户反馈
```

### 9.4 AI 职责

```text
1. 基于客户问题和引用知识生成回复草稿
2. 帮助客服减少组织语言和查资料成本
```

### 9.5 人工职责

```text
1. 审核 AI 回复草稿
2. 编辑不准确或不合适的内容
3. 点击发送回复
4. 对客户沟通内容负责
```

### 9.6 关键产品判断

客服回复不能由 AI 自动发送。原因是回复可能涉及故障判断、权限、费用、合同和服务承诺，必须由人工审核。

数据持久化必须优先完成。原因是真实产品不能把工单、回复、反馈只保存在前端或内存中，否则无法追踪服务进度，也无法做后续数据分析。

### 9.7 验收标准

```text
1. 客服可以发送回复
2. 客户可以看到客服回复
3. 客户可以确认客服回复是否解决问题
4. 客服发送回复后工单进入待客户确认状态
5. 不允许客服绕过必要步骤直接关闭工单
6. 客户在待确认期间直接继续输入时，系统应自动追加到当前工单
7. 客户补充说明后工单应回到处理中
8. 客服工作台能看到客户补充说明
9. 回复记录可以长期保存
10. 工单状态可以长期保存
11. 服务重启后仍能读取历史工单、回复和客户补充说明
12. 客户反馈可以被保存并关联到工单
```

### 9.8 阶段产出价值

```text
1. 主链路从“客户提交工单”补齐到“客服回复客户、客户确认结果”。
2. AI 回复草稿提高了客服处理效率，但最终回复仍由客服审核发送。
3. 工单状态增加“待客户确认”，更符合真实客服处理节奏。
4. 客户直接继续输入时默认补充到当前工单，减少客户理解工单流转的成本。
5. 工单、回复、客户补充说明、状态和客户反馈被持久化保存，支持后续追踪和统计。
6. 系统从演示态升级为具备基础业务数据沉淀能力的可用版本。
```

## 10. 阶段四：知识库管理后台

### 10.1 阶段目标

让运营和客服主管可以维护企业知识内容，使 AI 回答具备可运营、可优化的知识来源。

### 10.2 目标用户

```text
运营人员、知识库管理员、客服主管。
```

### 10.3 真实使用场景

运营人员根据客服反馈和数据看板发现某类问题经常未解决，于是补充或修改知识内容。

### 10.4 已实现功能

```text
1. 知识列表
2. 分类筛选
3. 状态筛选
4. 关键词搜索
5. 新增知识
6. 编辑标题、分类、标签、正文
7. 启用知识
8. 停用知识
9. 客户侧问答只检索启用知识
```

### 10.5 AI 职责

```text
AI 不负责维护知识内容；
AI 负责在客户提问时检索并使用已启用知识。
```

### 10.6 人工职责

```text
1. 维护知识内容
2. 判断知识是否准确
3. 决定知识是否启用
4. 根据未解决问题优化知识库
```

### 10.7 当前第一版简化逻辑

```text
一条知识 = 一个可检索知识块。
```

### 10.8 真实化优化项

真实产品更合理的逻辑是：

```text
上传长文档
↓
系统自动切分 Chunk
↓
生成向量索引
↓
运营审核和调整
↓
发布给 AI 检索使用
```

优先级：

```text
P2 真实产品能力优化。
```

计划位置：

```text
真实大模型接入 / RAG 优化阶段。
```

设计取舍说明：

```text
第一版先用结构化知识条目验证业务闭环；
后续再升级为文档上传、自动分段和向量检索，以接近真实企业知识库产品。
```

### 10.9 验收标准

```text
1. 运营可以新增知识
2. 运营可以编辑知识
3. 运营可以启用或停用知识
4. 客户侧 AI 只引用启用知识
5. 停用知识后不再被客户侧 AI 使用
```

### 10.10 阶段产出价值

```text
1. 企业知识从固定示例升级为可维护的后台内容。
2. 运营可以通过分类、标签、启停状态管理 AI 的知识来源。
3. 停用知识不会再被客户侧引用，降低错误知识继续影响回答的风险。
4. 知识库管理为后续 RAG、文档分段和向量检索升级打下基础。
```

## 11. 阶段五：数据看板

### 11.1 阶段目标

让产品经理和运营负责人判断 AI 是否真的提升了客户支持效率，并发现知识库和流程优化方向。

### 11.2 目标用户

```text
产品经理、运营负责人、客服主管、支持团队负责人。
```

### 11.3 已实现功能

```text
1. 总咨询量
2. AI 解决率
3. 转人工率
4. 知识命中率
5. 工单数量
6. 反馈数量
7. 启用知识数量
8. 停用知识数量
9. 客户反馈分布
10. 工单状态分布
11. 问题分类分布
12. 知识状态分布
13. 近期仍需帮助问题
```

### 11.4 AI 职责

```text
AI 本身不负责看板决策；
系统记录 AI 问答、知识命中、转人工和客户反馈数据。
```

### 11.5 人工职责

```text
产品经理和运营负责人根据数据判断：
1. 哪些问题 AI 已经能解决
2. 哪些问题需要补充知识
3. 哪些问题应该继续转人工
4. 哪些流程需要优化
```

### 11.6 关键指标口径

```text
AI 解决率：
客户点击已解决的反馈占比。

知识命中率：
AI 问答中成功命中知识库的比例。

转人工率：
AI 判断需要人工或客户提交工单的比例。

近期仍需帮助问题：
用于发现知识库缺口和回答质量问题。
```

### 11.7 验收标准

```text
1. 每次客户提问都能被记录
2. 看板能展示核心指标
3. 看板能展示反馈、工单、分类和知识状态分布
4. 看板能展示近期仍需帮助问题
5. 指标口径与业务流程一致
```

### 11.8 阶段产出价值

```text
1. 系统具备了评估 AI 服务效果的基础指标。
2. AI 解决率、知识命中率和转人工率可以反映当前服务效率。
3. 近期仍需帮助问题可以反向指导知识库补充和流程优化。
4. 数据看板让项目从“功能可用”进入“效果可评估”的阶段。
```

## 12. 阶段六：真实大模型接入

### 12.1 阶段目标

将当前模拟回答升级为真实大模型回答，形成更完整的 LLM + RAG 能力。

### 12.2 目标用户

这是系统能力阶段，间接服务所有角色：

```text
客户获得更自然的回答；
客服获得更好的总结和回复草稿；
运营维护的知识可以影响模型输出；
产品经理可以评估真实模型效果。
```

### 12.3 已实现功能

```text
1. 新增 Seed 2.0 Pro 调用服务
2. 支持通过环境变量配置方舟 API Key、接口地址和 Seed 2.0 Pro 对应的模型 ID 或推理接入点 ID
3. 客户侧问答优先使用 Seed 2.0 Pro 生成回答
4. 客服工作台回复草稿优先使用 Seed 2.0 Pro 生成
5. 将检索到的知识片段注入 Prompt
6. 要求模型只基于知识库回答
7. 大模型失败时降级到规则兜底回答或转人工提示
8. 无知识命中时不调用模型编造答案
```

### 12.4 AI 职责

```text
1. 基于知识库生成自然语言回答
2. 基于上下文生成客服回复草稿
3. 在知识不足时承认无法确认
```

### 12.5 人工职责

```text
1. 配置 API Key
2. 审核 Prompt 效果
3. 审核知识内容
4. 对客服最终回复负责
```

### 12.6 验收标准

```text
1. 配置有效 API Key 后，客户侧回答来自 Seed 2.0 Pro
2. 回答仍然基于知识库
3. 回答保留引用来源
4. 无知识命中时能兜底
5. 大模型接口失败时项目仍可演示
6. 客服回复草稿可以由 Seed 2.0 Pro 生成
```

### 12.7 阶段产出价值

```text
1. 系统从模拟回答升级为真实大模型生成回答。
2. RAG 流程把企业知识注入模型输入，减少模型脱离业务资料回答。
3. Prompt 约束明确了回答范围、引用要求和无知识命中时的处理方式。
4. 模型失败兜底机制可以保证客户问题不会因接口异常而丢失。
5. 真实模型接入后，数据看板可以继续评估模型回答效果。
```

### 13.4 第一版上线方案

```text
第一版采用单服务部署方式：
1. 一个 Node 服务同时提供前端静态页面和后端 API
2. SQLite 数据库通过持久磁盘保存
3. 大模型 API Key、模型 ID、接口地址通过服务端环境变量配置
4. 客户侧、客服工作台、知识库后台和数据看板通过同一个域名的不同路径访问
```

采用该方案的原因：

```text
1. 当前阶段重点是验证 AI 客服业务闭环，而不是建设完整企业生产架构
2. 单服务部署能更快验证客户提问、AI 问答、工单流转、知识库维护和数据看板
3. 当前系统使用 SQLite，适合第一版 MVP 试点，但需要持久磁盘保证数据不因服务重启丢失
4. 环境变量可以保护大模型 API Key，不把密钥写入前端代码
```

### 13.5 生产化升级路径

如果系统进入正式生产环境，建议升级为：

```text
1. 前端独立部署到 CDN、对象存储、Vercel 或 Netlify
2. 后端独立部署到云服务器、容器平台或托管 Web Service
3. SQLite 替换为 PostgreSQL、MySQL 或其他托管数据库
4. 增加登录权限、角色权限和企业数据隔离
5. 增加日志、监控、告警和接口错误追踪
6. 增加 Secret Manager 或云平台密钥管理
7. 增加数据备份和恢复策略
8. 增加灰度发布和回滚机制
```

产品取舍说明：

```text
第一版上线方案是真实可访问的 MVP 部署方式，用于验证核心业务闭环和产品体验；
它不是中大型企业正式生产环境的最终架构。
```

## 19. 项目构建方法总结

这个项目不是先堆 AI 功能，而是按真实产品构建顺序推进：

```text
先完成客户问题能不能被回答；
再完成 AI 解决不了时能不能转人工；
再完成客服能不能回复客户；
再完成数据能不能保存；
再完成知识能不能运营；
再完成效果能不能评估；
再接入真实大模型；
最后完成上线和交付说明。
```

这体现了 AI 产品系统建设中最重要的原则：

```text
不是先堆模型能力，而是把模型能力放进真实业务流程里，并设计清楚边界、兜底、指标和迭代路径。
```
